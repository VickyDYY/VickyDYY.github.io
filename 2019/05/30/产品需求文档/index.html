04产品需求文档

Posted on 2019-05-30 | In 产品学习笔记

周一广联达失败了后，做了两天的酸菜鱼，今天又恢复的元气，还有要继续努力学习呀！就算是一条酸菜鱼也不能放弃自己的梦想呀！每天起床第一句，先给自己打个气！

今天主要是进行PRD相关知识的复习，本节笔记主要对PRD的整体框架和某些部分的概念进行的整理，后续PRD每一部分的技能需求还未涉及，未完待续……

一、What（什么是PRD？）

PM三大文档：

  • BRD（商业需求分析文档）
  • MRD（市场需求分析文档）
  • PRD（产品需求分析文档）

PM重点放在PRD，PRD就是一份产品说明书，包括：产品界面、功能、业务逻辑

PRD作用：

1、加强内部沟通：明确产品需求、产品细节、实现结果

2、存档：产品迭代记录，有据可查，交接容易

二、How（PRD撰写步骤）

1、前期准备工作

1.1 基本信息：
  • 新产品：产品名称+功能名称，例：嘀嘀打车安全服务功能
  • 迭代产品：功能名称+迭代历史，例：微信”抢红包”功能第15版迭代
1.2 需求背景+解决方案：
  • 需求背景：
    • Who？（boss？业务？用户？）
    • 场景？（例：嘀嘀打车时用户打车应单率低）
    • 遇到的问题？（例：用户希望打车能够快速应答）
  • 解决方案：（例：给用户同时派发多个车型）
    • 需求分析和需求处理的数据（多少用户）
    • 用户访谈，用户调研
    • 竞争对手、竞品采取的方式

2、正式撰写PRD

2.1 需求描述+对应功能说明：
  • 产品功能列表：优先级排序
  • 产品功能描述：解决或优化哪些问题？
  • 产品功能结构图：梳理模块、功能之间逻辑，将功能列表进行归类

例：美团外卖配送系统

场景：某二、三线城市，美团外卖应单率不理想

解决问题：提升应单率

功能列表：（+优先级）

  • 功能1（P1级）：针对自营骑手，骑手派发应率的优化（从系统角度保证应单合理）

  • 功能2（P2级）：自营骑手无法应单转美团众包骑手

  • 功能3（P3级）：自营和众包骑手均无法应单，推动商家进行第三方配送（嗒嗒、蜂鸟）

功能描述：（解决优化的问题）

  • 目前自营骑手处于资源闲置的情况，做派发的一类算法，提升自营骑手应单效率同步解决用户需求，
  • 解决自营骑手不愿意送的偏远单或大额订单
2.2 业务流程：
  • 产品整体业务流程：业务流程图（新产品）
  • 迭代功能的业务流程：业务流程图（迭代产品）

（LOC商品：线上买，线下扫码核销，类似美团的很多商品）

三、实操案例

策略类产品：风控策略

需求背景：

某平台推出预定返现活动后，检测到某些用户套现补贴，只预定，不支付金额的情况，需要针对这个场景进行控制，不能干扰到正常用户下单，又能有效的防范

需求提出：
  • 用户特征的提取
  • 套现方法路径的观测
  • 订单集体特征的观测
  • 方案实施
发现特征：
  • 号码段67%为170号段
  • 用户特征对历史订单不超过1笔（新用户/次新用户）
  • 历史订单金额在10元以下，正常用户均值在30元
提出策略：
  • 170号段的用户将采用前付的方式
  • 历史订单数量N<=1的，先付全款，再领补贴
  • 历史金额在20元及以下的订单，不在N的取值范围内

四、课后练习

【作业目标】

产品经理在提出需求时，往往要通过一份标准的产品需求文档来向开发、设计、测试等人员传递信息，明确如下几个问题：需求方是谁？在什么场景下？遇到什么问题？解决的方案是怎样的？一些产品新人对「场景」这个词可能不太理解，那么「场景」和「问题」又有什么区别呢？

【作业题目】

通过询问、查阅资料等方式，以某一产品为例，撰写一份可以区别“场景”和“问题”这两个概念的说明文。

【参考答案】

以「小红书」为例，我们来区别一下“场景”和“问题”这两个概念：

首先，在小红书中体现场景的例子有：

  • 后天是情人节，我要和女友出去吃饭

  • 双十一来临，碰巧香水用光了，我在考虑重新买一瓶

  • 我是一名美妆博主，有很多关于美妆的干货可以分享

其次，在小红书中体现问题的例子有（句子中加粗的部分）：

  • 后天是情人节，我要和女友出去吃饭，但是不知道该选一家怎样的餐厅，让女友满意。（发现）

  • 双十一来临，碰巧香水用光了，我在考虑重新买一瓶，香水的种类太多，不知道如何选择。（决策）

  • 我是一名美妆博主，有很多关于美妆的干货可以分享，我需要一个可以记录生活点滴和美好片段的平台，最后达到盈利的目的。（记录）

总结一下：

  • 场景：是人物所在环境的各类客观事物的集合，包括时间、空间等，5W1H的组合，谁（who）因为什么（why）在什么时候（when）在哪里（where）正在做（what），状态如何（how），更严格的要求还应考虑场景流，即从哪个场景来，下一步要到哪个场景中去。

  • 问题：是人物的痛点，在场景中可以理解为，客观事物反映到人物身上，再回馈出来的情绪/痛点。

尽管以上的场景和问题拆解的不够详细，但可以看出「分析问题时，必须依托于场景」。场景要能触发人的反应，才会产生需求/痛点。
